Concept Architecture

The skeleton of Tendd: what entities exist, what screens exist, what job each one serves, and how a person moves between them. Structure only, not screen content.

Phase IA. Prompt 1 built the skeleton (entities, sitemap, navigation, flows, coverage). Prompt 2 adds the detail Wireframes needs: global components documented once, and a page card per screen with ordered content blocks and real states. The skeleton is locked. Every block traces to a job in JTBD or a locked decision (D1 to D5). Full source: docs/sitemap.md and docs/flows.md.

Flag raised against the locked skeleton (for Wireframes)

The "Save" tab has no dedicated landing screen. Navigation maps Save to a "Cancel hub", but no screen by that name exists in the locked 16. The cards treat the Save tab as landing on Home filtered to its cancel-candidates view. For Wireframes: either land Save on that Home view, or add a Save/Review screen in a later structural pass. Not fixed here because the screen list is locked.

A second, minor note: Connect Bank and Add Subscription are reused by Connections / Accounts to add a source later, consistent with the existing reuse pattern. Not a defect.

Step 1

Entities

The core objects a person deals with to close their jobs. Screens display entities; entities grow out of jobs, not out of a competitor's menu. Nine core entities, plus objects held under question.

1. Subscription

Name, real merchant name and logo, amount, frequency, next date, category, status, source, linked account. The central object.

J-MAIN, J3

2. Account (source)

Bank connection via Plaid, or a manual private source. Status, last sync, count detected.

J1, J5, D5

3. Preset

Service template (name, logo, typical price, frequency) so manual entry is low effort. 400+ at launch.

J5, D2

4. Alert

Price change and payment failed are free; trial ending, unusual, duplicate are Pro. Plain-language, active voice.

J4

5. Charge

A billing instance: date, amount, subscription, status. Full timeline is Pro history.

J4, D3

6. Cancellation

Method (basic free, full guide Pro), progress status, money saved. The relief moment stays free.

J2, E2

7. Snapshot

Shareable card: count, monthly total, optional win, no bank data. Virality is a hypothesis.?

S1, E2

8. User & data controls

Minimal identity, path, plan, notification preferences, data-access explanation, one-tap deletion.

E3, J1

9. Plan (Tendd Pro)

Tier (Free / Pro monthly / yearly / lifetime), price $7.99 or $69, status. A business rule.

D3, D4

Under question (no entity, no screen): household / shared view JH1, Gmail scan and screenshot import v2, AI assistant JH2, budgets, credit score, investments, bill negotiation (all explicitly cut). A scope note: this concept follows locked decision D3 (free tier is unlimited on subscription count) over the older CLAUDE.md "up to 10" line.

Sixteen screens across six clusters, derived from the entities and from what each person is trying to do. Every screen names the job it serves. Amber marks Emma's primary path. Empty, error, and loading are states, shown in the flows, not screens.

Step 3

Navigation

A four-item bottom tab bar (thumb reach on a phone), deliberately small so the product stays calm. Each tab is an entry into one job cluster.

TabOpensJob clusterWhy it earns a global slot
HomeSubscription ListJ-MAIN + E1The reason the product exists. Reachable in zero taps, so it is the default.
AlertsAlerts / ActivityJ4Also the return engine. Without an external hook this segment does not come back (BP4).
SaveCancel hubJ2 + E2The emotional peak and upgrade trigger. Claudia's primary job gets a calm door.
YouSettings / ProfileE3 + J1Trust is ongoing. Ravi needs one-tap data control always findable.

Depth to the main job (J-MAIN, see all recurring charges calmly):

3 taps in the first session: Welcome, Path Choice, Connect Bank, then Guided Reveal. At the ceiling, not over it.

0 to 1 tap in every session after: the app opens on Home. The steady state is effectively zero depth.

History and Trends does not get a tab (it is Pro-gated, and a mostly-locked tab would break calm). Upgrade is not a tab either; it appears at the gate, in context.

Step 6

Critique

Four defect classes, most dangerous first. Fixes are proposals for Prompt 2, not applied here. All dead-end and missing-state fixes are states and copy on existing screens, so the 16-screen inventory does not change.

1. Dead ends

WhereDefectProposed fix
Connect Bank (Flow A)Declining both retry and manual fallback leaves the user with no list.Keep an "add them yourself" action persistent on the error state; never an error with only a dead exit.
Add Subscription (Flow B)Ravi abandons manual entry half-done and leaves with a partial list.Persist partial progress, run the reveal on a partial list, and re-hook with the return notification.
Cancel Guide (Flow C)A blocked external cancellation leaves the user stuck with no next step.Add a "could not cancel" state with alternative steps or a mark-to-retry, so there is always an in-app move.
Subscription Detail (Flow D)A failed-payment alert ends at "informed" when the fix is at the bank or merchant.Add a plain-language next-step card ("update your card" or "no action needed, retries on DATE").

2. Missing states

WhereDefectProposed fix
Home / Subscription ListEmpty, loading, and sync-error states are not yet specified.Define a calm empty state, a non-alarming loading state, and a sync-error state that keeps the last known list.
Subscription DetailNo state for when enrichment fails and the name stays cryptic. This threatens J3.Define an "unrecognized charge" state that invites naming or categorizing it.
History and TrendsA paid user with under 3 months of history has no defined empty state.Define a "still gathering your history" state, distinct from the Pro upsell.
Cancel GuideNo state for a service that has no guide yet.Define a "no guide yet" state with generic steps and a request-a-guide action.
Share SnapshotCard generation and share failure are not specified.Define a loading state while the card renders and an error state if the share fails.

3. Excess depth

No violation. First-session J-MAIN sits at exactly 3 taps, at the ceiling. Watch item: do not add any onboarding step, or the main job moves to 4 taps. Steady-state J-MAIN is 0 to 1 tap. J4 from a notification is about 2 taps; J2 for Claudia is 2 taps to the guide.

4. Orphans

One orphan screen: Upgrade / Tendd Pro, retained as a contextual gate justified by D3 and D4. No orphan jobs. Every job in JTBD is served by at least one screen.

Prompt 2, Step 1

Global Components

Elements that repeat across screens, documented once. Page cards reference these by name and do not redescribe them.

GC1. App Header

  • 1. Back or menu control
  • 2. Screen title
  • 3. One contextual action slot (add, share, or edit)

Mobile: top bar. Desktop: merges into the top of the left rail (structural, see Home and the deltas below). States: signed-in, onboarding (no tab bar), no-account-yet [?]. Traces: navigation chrome, auth-state axis (Ravi).

GC2. Global Tab Bar (primary navigation)

  • 1. Home
  • 2. Alerts
  • 3. Save
  • 4. You

Mobile: bottom tab bar. Desktop: left rail, same four destinations (structural delta). Hidden across the whole onboarding chain. Traces: Navigation section, one job cluster per tab.

GC3. Recurring Summary Strip

  • 1. Subscription count
  • 2. Monthly total (the largest number on screen)
  • 3. Optional context line, "what you have signed up for"

Same content on both. States: populated, loading, empty (an invite, not a bare zero). Traces: J-MAIN, D1, the "biggest number" design principle, Subscription aggregate.

GC4. Subscription List Item

  • 1. Merchant logo
  • 2. Real merchant name
  • 3. Amount and frequency
  • 4. Next billing date
  • 5. Status tag

Layout differs (stacked on mobile, row on desktop), content same. States: normal, unrecognized (name still cryptic, invites naming), cancelled. Traces: Subscription entity, J-MAIN, J3 (M3).

GC5. Alert Item

  • 1. Alert type
  • 2. Plain-language, active-voice message
  • 3. Related subscription
  • 4. One suggested next action

Appears as a banner on Home and a row in Alerts. States: free type (price change, payment failed), Pro type (trial, unusual, duplicate) with a gate. Traces: Alert entity, J4, M1, D3.

GC6. Data Source and Trust Indicator

  • 1. Current source (bank or private/manual)
  • 2. Read-only, cannot move money reassurance
  • 3. Link to the full "what we read" explanation

Same content on both. States: bank-connected, private/manual, needs-reauth. Traces: Account and User controls entities, J1, E3, M2, D2, D5.

GC7. Pro Gate and Plan Chip

  • 1. Plan indicator (Free or Pro)
  • 2. At a gate, a short "what unlocks" line and an upgrade action

Same content on both. States: free (upgrade affordance at gates), Pro (badge, no gate). Traces: Plan entity, D3, D4. Carries the one decision-justified screen (Upgrade) into context.
